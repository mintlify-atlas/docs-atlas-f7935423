Once How is installed, you interact with it through a single slash command: `/how`. You type a question about your codebase, and How handles everything else — complexity assessment, agent orchestration, and synthesis. This page walks you through the full experience with real example prompts.

## Type your first question

Open a Cursor agent session in your project and type any of these example prompts to get started:

```text
/how How does message virtualization work?
```

```text
/how Walk me through what happens when a user sends a message
```

```text
/how How does the auth middleware check permissions?
```

```text
/how How is the auth service structured? Also critique the design.
```

<Tip>
  You don't need to phrase questions in any specific format. How parses natural language — subsystem names, feature descriptions, runtime traces, and architectural overviews all work.
</Tip>

## What happens after you press Enter

<Steps>
  <Step title="How reads your question and assesses complexity">
    How parses what you are asking about and decides between two execution paths:

    - **Simple path** — a single module, a narrow utility, or a focused question like "How does the auth middleware check permissions?" How runs end-to-end in a single agent with no fan-out.
    - **Complex path** — a subsystem spanning multiple files or services, a cross-cutting feature flow, or a broad architectural overview. How decomposes the question into 2–4 parallel exploration angles and spawns explorer subagents.

    When in doubt, How leans toward the simple path and can escalate if the explainer hits a wall.
  </Step>
  <Step title="Explorer agents trace the codebase (complex questions only)">
    For a complex question like "How does message virtualization work?", How might spawn three parallel explorers:

    - **Explorer 1** — the data model and state management
    - **Explorer 2** — the rendering pipeline and DOM interaction
    - **Explorer 3** — the scroll and measurement infrastructure

    Each explorer starts broad (globbing directories, grepping for key types and interfaces), then follows the thread — tracing call chains, reading actual code, and noting anything non-obvious or surprising. Explorers run in parallel, so their combined runtime is roughly the cost of one.
  </Step>
  <Step title="A synthesis agent produces the explanation">
    Once the explorers return their findings, a synthesis agent reconciles them into a single coherent explanation. The output follows a consistent structure:

    - **Overview** — what this thing is and why it exists
    - **Key Concepts** — the important types, services, and abstractions
    - **How It Works** — the core flow from trigger to effect, in prose
    - **Where Things Live** — a map of the relevant files and directories
    - **Gotchas** — non-obvious things, sharp edges, and historical context

    Not every section appears for every question — the agent adapts the structure to what makes sense.
  </Step>
  <Step title="You receive the explanation">
    How presents the synthesis agent's output directly. You can read the explanation, follow the file references into your codebase, and ask follow-up questions.
  </Step>
</Steps>

## Simple path vs. complex fan-out

<AccordionGroup>
  <Accordion title="When does How take the simple path?">
    How takes the simple path when your question is narrow enough that a single agent can explore and explain without missing important context. Examples:

    - "How does the auth middleware check permissions?" — a focused question about one piece of middleware
    - "How does function X work?" — a specific function
    - "What does this utility do?" — a small, self-contained module

    The simple path is faster and produces the same output format. How defaults to it when there is any doubt.
  </Accordion>
  <Accordion title="When does How fan out to parallel explorers?">
    How fans out when the question spans multiple files, services, or architectural layers — when one agent tracing a single thread would miss important parts of the picture. Examples:

    - "How does message virtualization work?" — involves data models, rendering, and scroll infrastructure
    - "Walk me through what happens when a user sends a message" — crosses service boundaries
    - "How is the auth service structured?" — a broad architectural overview

    How spawns 2–4 explorers depending on the question. Narrow complex questions get 2 explorers; broad subsystem overviews get up to 4.
  </Accordion>
</AccordionGroup>

## Triggering critique mode

Add any indication that you want architectural analysis to your question, and How will run the full explain flow first, then spawn three independent critics across different AI models.

```text
/how How is the auth service structured? Also critique the design.
```

After the explanation, you receive a lead-judgment verdict that categorizes each finding:

| Category | Meaning |
|----------|---------|
| **Act on** | Architectural problems worth fixing now |
| **Consider** | Real concerns where cost/benefit is unclear |
| **Noted** | Valid observations, low priority |
| **Dismissed** | Wrong, missing context, or style preference |

The explanation always appears first and stands on its own. You don't need to read the critique section if you only wanted to understand the system.

<Info>
  Critique mode spawns three critics in parallel across different models. This takes longer than explain mode but produces a richer analysis because each critic works independently before the lead-judgment pass reconciles their findings.
</Info>
